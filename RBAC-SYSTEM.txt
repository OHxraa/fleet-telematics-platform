🔐 HOW IT WORKS BEHIND THE SCENES:
═════════════════════════════════════════════════════════════════════════════

When Paul (ADMIN) tries to edit a vehicle name:

1. Request comes in: PUT /api/vehicles/xyz { name: "New Name" }
2. Middleware extracts user: Paul (id: xyz)
3. Database lookup: What role is Paul? → ADMIN
4. Permission check: Does ADMIN have "vehicle.edit_name"? → YES
5. Resource check: Any data ownership issues? → NO (admins see all)
6. ✓ REQUEST ALLOWED - Vehicle name updated
7. Audit log: "Paul changed Trailer 001 name to Trailer A"

When Driver John tries to view Sarah's vehicle:

1. Request: GET /api/vehicles/sarahs-trailer-id
2. Middleware: John (id: xyz, role: driver, assigned_vehicle: johns-trailer)
3. Permission check: Does driver have "vehicle.view"? → NO (only view_own)
4. Resource check: Is this John's assigned vehicle? → NO (it's Sarah's)
5. ✗ REQUEST DENIED - 403 Forbidden
6. Audit log: "John attempted unauthorized access to sarahs-trailer-id"
